AXIS PRE-IMPLEMENTATION REVIEW · JULY 2026

Full Design-System & UX Review

Scope: Axis Layout System.dc.html (shell, 5-region dock engine, widget zones, edit mode) plus AxisWidget, AxisGroup, AxisGridPanel, AxisBlockEditor, AxisPresetBrowser, AxisFcPanel. Reviewed against live behavior at 1440 / 924 px and the Tablet profile. No feature is removed by any recommendation.

1 · Executive Summary

The architecture is genuinely strong: a real tiling dock engine (5 regions, per-slot tabs, cell splits, collapse strips), a widget system with 3 size states, cross-instance buses for split panels, and per-profile layouts. That skeleton should not change. What blocks implementation-readiness is a layer of gaps around it: the size-state auto-fit is only wired to custom panels (so the top bar clips at ≤1100 px), the Signal Grid sizes itself against the whole content area instead of its own pane (so docking anything shrinks it into scrollbars), five of seven primary nav items do nothing on the default desktop layout, there is no keyboard/focus/ARIA layer at all, and app state (saved / dirty / connected / syncing) is simulated with toasts rather than modeled.

FIX FIRST
Top-bar widget auto-fit · Grid pane-relative scaling · dead nav sections · edit-ribbon dropdown anchoring · tablet-profile clipping.
HIGH IMPACT
Real dirty/save/sync state model · keyboard + Escape layer · touch-target expansion · unify the two docking systems · clickable History.
KEEP AS-IS
Region engine + reflow invariant · widget/panel/param model · edge-drop panel creation · buses · Customize FAB entry · default panel layouts.

2 · Major UX Risks

R1
Layout breaks below ~1100 px. Verified at 924 px: the preset widget truncates to “DEB…”, the view switch clips mid-chip, and the top bar overflows. The 3-size auto-fit contract exists (mkW + estW) but this._panW is populated only for custom panels — tl/tc/tr, bottom, and gridbar zones never shrink.
R2
Docking any panel cripples the Signal Grid. Verified: opening Signal Chain (bottom) leaves the grid pane ~110 px tall with horizontal + vertical scrollbars and only 1.5 rows visible. Grid cell math uses contentEl.clientWidth/Height (the whole body) instead of the pane’s own availW/availH. Violates the “Auto mode never scrolls” expectation and the no-h-scrollbar invariant.
R3
Primary navigation is dead on the default desktop layout. Verified: clicking Sets highlights the rail item and nothing else happens. onNav only sets section; section stubs render only when contentMode==="pages". Ctrl / Scn / Live / Sets / Setup are silent no-ops. Beginners will conclude the app is broken.
R4
No state model behind Save. There is no dirty flag anywhere: parameter swipes, scene changes, and layout edits all report success via toast, but nothing tracks unsaved work, and preset recall silently discards edits. For a device editor this is the single largest data-loss risk.
R5
Zero keyboard accessibility. Every control is a div onClick; there is no tabindex, no focus style, no Escape-to-close on any flyout/menu/drawer, and the advertised ⌘K shortcut on “Add block” is not implemented. Power users — the target audience — cannot drive the app without a mouse.
R6
Two competing docking systems. The legacy gridDock/editorMode (drawer/right/left/floating) states coexist with the region engine’s dock.place. Layout presets set one, the Options menu sets the other, and gridInMain reads the legacy one — producing states where the two disagree (grid “minimap” logic vs. grid pane).
R7
Edit-mode chrome is not touch-usable. Hide “×” is 17–19 px, pane header buttons 22 px, scene/FC chips 20–22 px tall, resize handles 6 px. On the Tablet profile — the profile that exists for touch — most customization chrome is below half the 44 px minimum.

3 · Design-System Findings

3.1 Typography

  • Eleven ad-hoc sizes between 8 and 16 px (8, 8.5, 9, 10, 11, 12, 12.5, 13, 13.5, 15, 16). Consolidate to a 5-step scale: 10 (mono meta) · 11 (mono labels) · 12 (UI) · 13 (body/controls) · 15 (panel titles). Kill 8/8.5/9 entirely — they exist only in rail labels, zone hints, and widget captions and are illegible on 1×-density screens.
  • Role convention is good (JetBrains Mono = machine values/labels, Hanken Grotesk = human UI) but applied inconsistently: pane tab labels are Grotesk 10 px bold while ribbon section labels are Mono 9 px. Rule: mono for values + tracking-spaced section labels at ≥10 px; Grotesk for anything clickable at ≥11 px.
  • Preset name in the top-left widget truncates with no tooltip/fallback (“DEB…”). Add title + min readable width before ellipsis, and prefer dropping the PRE arrows to compact state before eating the name.

3.2 Color & contrast

  • --textfaint #6e6e78 on #0e0e10 ≈ 3.9:1 — fails WCAG AA for the 9–10 px labels it is used on. --textmuted #56565e ≈ 2.7:1 fails everywhere it carries meaning (timestamps, zone names, widget hints). Fix by policy, not per-spot: textfaint → #8a8a94, textmuted reserved for genuinely decorative ghost text (DROP HERE hints may stay dim — they are edit-mode wayfinding).
  • Category color ramp on grid blocks (shade −0.42 → −0.62 + colored border) is excellent and should be documented as the canonical “tinted block” recipe. The white 8 px channel badge on colored blocks passes; keep.
  • Accent duplication: teal appears as #35c9d6, #4fd1dc, #8fe3ea plus bg tints #101d1e, #12262a, #13282a, #1c2b2c and borders #234d4f, #2f6d70, #1a3a3c. Name them (accent / accent-bright / accent-text / accent-bg-1..2 / accent-border-1..2) so future widgets stop inventing near-duplicates.

3.3 Spacing, radius, controls

  • Gaps use every integer 2–14. Snap to a 4-px scale (4/8/12/16) with 2 allowed only inside segmented controls. Radii use 5,6,7,8,9,10,11,12,14,22 — collapse to 6 (chips) · 9 (controls) · 12 (panels/cards) · pill.
  • Four segmented-control implementations exist with different paddings and heights: seg() (ribbon), pill() (AxisWidget), gmSeg() (grid mode), chipBase (panel chips). Define one spec — 28 px height, 6-px radius items in an 8-px radius track, active = accent fill + #06181a text — and reuse it in all four places.
  • Buttons have hover styles but no pressed/active state and no disabled convention (size steppers fake disabled with color only; cursor stays pointer on some). Define: disabled = 40% text, no border, cursor default, aria-disabled.

3.4 States & iconography

  • Three different collapse affordances coexist: pane “–/+”, whole-dock “«”, and collapsed region strips. Keep all three behaviors, but give them one icon language (chevrons oriented to the region) and tooltips that name the target: “Collapse panel”, “Collapse LEFT dock”.
  • Unicode glyph icons (⠿ ✥ ▤ ⇲ ⊟ ‖ ⤢) render at inconsistent optical sizes across platforms. Acceptable for the prototype; for implementation swap to a single icon set at 16/20 px. Flag: ⇲ is used for both “density” and “resize” meanings.
  • Tooltips rely on native title except the param widget’s custom .axtip. The custom one (source-block tooltip) is the right pattern — promote it to the shared tooltip for all icon-only controls, since title is invisible on touch.

4 · Workflow Findings

W1 · Browsing & loading presets
The quick flyout works, but the full Preset Browser (Sources / List / Info panels) is reachable only through Edit Layout → Panels — a customization menu, not a daily-driver path. The hidden-by-default Search widget is the only other route. Change: make the Sets nav item open the Preset Browser (dock plist to main + psrc left + pdet right, same as Single Page does), toggling back on second press — exactly the pattern already proven by the FC nav item. Keep the flyout for one-tap recall. Add an unsaved-changes guard before recall (see W4).
W2 · Nav sections Ctrl / Scn / Live / Setup
Silent no-ops on the default desktop layout (stubs render only in “pages” content mode). Until their panels exist, each must at minimum produce feedback: open its stub as a panel in main (region engine already supports arbitrary panes) with a proper empty state — “Scenes manager lands here · use the Scenes widget meanwhile”. A nav item that visually activates but changes nothing trains users to distrust the whole rail.
W3 · Editing block parameters
Tap-to-open editor + swipe-to-adjust on grid blocks is a great power gesture, but it is fully undiscoverable, and two hidden gestures ride on it (double-click = reset to default; horizontal axis is captured then does nothing — dead code that also blocks horizontal pan on touch). Change: (a) first successful swipe shows a one-time hint toast (“Swipe ↑↓ adjusts <param> · double-tap resets”); (b) show the param name + value chip during hover on desktop, not only mid-swipe; (c) either implement the horizontal axis (channel A/B/C/D scrub would be natural) or stop claiming it so touch scroll passes through; (d) param knobs in the Block Editor already drag out as pinned widgets — add the same affordance hint on long-press.
W4 · Saving (preset) & dirty state
Save is a stateless toast. Change: introduce a real dirty flag set by param edits, scene changes, block toggle/remove, and channel switches. Surface it in the existing Save widget (amber dot + “Save*”), guard preset recall and Scene-load with a 3-option dialog (Save & load / Discard / Cancel), and support ⌘S. The Save widget already lives in the widget system, so this is a widget-state change, not new chrome.
W5 · Undo / redo & History
The Undo/Redo widget is hidden in every default layout, ⌘Z is unbound, and History rows (panel and drawer) are display-only — no hover state, no click, no “restore to here”. Change: bind ⌘Z/⇧⌘Z globally; make History rows interactive (click = preview highlight, “Restore” button on hover/focus with confirm); keep the widget optional but list it prominently in the Library under a “Safety” group.
W6 · Customizing a layout
Entry (FAB) and exit (Done + confirmation toast naming the profile) are solid. Friction points: (a) the Panels/Options dropdowns are hard-anchored at fixed top:106px right:150px — when the ribbon wraps to two rows (any width < ~1180 px) they detach from their buttons and can cover them; anchor to the trigger’s rect instead. (b) “＋ Panel” always creates the panel in RIGHT with no choice — either honor the drop position (edge-drop already exists and is better) or show a 5-region mini-picker. (c) Panel rename is dblclick-on-tab + window.prompt() — undiscoverable and unthemed; replace with inline editing via the existing right-click menu’s Rename. (d) ribbon “＋ Save” and top-bar “Save” both exist while layout auto-saves on Done — clarify: ribbon button becomes “Save as layout…” (named snapshot), auto-save note stays in the Done toast.
W7 · Add block (⌘K)
The widget advertises ⌘K but both the shortcut and the palette are unimplemented (click → toast). Ship the palette (searchable block-type list, category-tinted rows reusing the CAT colors, Enter inserts at first empty cell / selected cell) and register the real ⌘K binding — or remove the badge until it exists. Empty grid cells should also accept click → same palette (they currently render a “+” cursor with no handler payload).
W8 · Device & cloud state
Connection state is a 9-px green dot with a tooltip at the rail’s bottom; there is no disconnected, syncing, or write-in-progress representation, and Account (“cloud sync”) is a toast. Change: make the connection indicator a proper status widget (it already fits the widget system) with 4 states — connected / syncing (pulse) / write-pending (amber) / disconnected (red + reconnect action) — placeable anywhere, defaulting to the rail. Device writes and cloud sync must show progress + failure with retry, not just success toasts.

5 · Mobile & Responsive Findings

M1
Top bar overflow (verified at 924 px). Wire the auto-fit pipeline to every zone: measure tl/tc/tr/bottom/gridbar container widths (extend the _panW mechanism or a per-zone ResizeObserver), sum estW of members, and step widgets default→compact→mini until the zone fits. Overflow-priority order per zone (e.g. tr sheds status group first, Save last) should be part of the widget descriptor.
M2
Tablet profile clipping (verified). In the 1024-px device frame the grid pane collapses to ~1.5 rows behind a horizontal scrollbar and the Signal Chain’s second row is cut by the bottom bar. Same root cause as R2 plus fixed default region sizes (bottom 300 px) that don’t scale to the 760-px frame — clamp region defaults to percentages of frame height (bottom ≤ 34%).
M3
Nav-hide “×” badges render detached below bottom-bar nav items in edit mode (absolute top:-5px against a wrap that has no height there). Anchor edit chrome inside the item’s box (top-right inset) so it can never orphan, and enlarge to 24 px visual / 44 px hit.
M4
Two device concepts collide. mobVP (viewport < 760 px → hamburger + overlay rail) and profile==="tablet" (bottom nav) are independent; on a real narrow tablet both trigger, yielding hamburger and bottom nav simultaneously. Rule: profile decides nav placement; mobVP only decides overlay-vs-inline for the side rail, and is suppressed when navMode is “bottom”.
M5
Touch targets. Minimum 44-px hit areas via invisible expansion (padding/pseudo-hit rect), keeping visuals: scene chips (21×22), FC layout chips (20×22), steppers (22×24), pane mini-buttons (22), edit ctrl (19), nav hide (17). Region/cell resizers 6 px → 24-px hit zone with the visible 2-px line unchanged. Swipe-adjust on blocks: on touch, require a 250-ms press before arming the gesture so page scroll wins by default.
M6
Bottom-sheet opportunities. On tablet, the Block Editor as a bottom drawer already exists — extend the same treatment to the preset flyout (currently a 400-px right sheet, fine) and the Widget Library (320-px right sheet is narrow for touch: 88vw bottom sheet on tablet). Dialogs (future confirm dialogs from W4) should be bottom sheets on touch profiles.

6 · Visibility, Feedback & State

  • Toast system: single slot; flash() clears the previous timer, so rapid actions (drop + auto-panel + group) silently swallow messages. Queue up to 3 stacked toasts; give destructive/system toasts an inline action (“Panel removed — Undo”). Never deliver instructions by toast (current: “drag it anywhere”, “drag widgets in”) — instructional hints belong on the target (empty-panel hint already does this correctly).
  • Drop rejection before the drop: “Too wide for the dock” fires only on release. During drag, tint the right dock red with the same message so the user aborts early. Better: since widgets have compact/mini states, auto-shrink instead of rejecting — rejection contradicts the auto-fit contract.
  • Edge-drop affordance: auto-creating a panel from an edge drop is invisible until it happens. While dragging a widget, render the same dashed region previews used for pane drags (“NEW PANEL · LEFT”) when inside the 15–18% edge bands.
  • CPU meter: static value, no thresholds. Add amber ≥ 75%, red ≥ 90% (bar + value), tooltip “DSP load”. It’s the device-safety metric guitarists actually watch.
  • Scene/preset context: the chain label pattern (“DEBUG · Scene 2”) exists but isn’t shown anywhere persistent when the FC/Chain panel is closed. The preset widget should carry the scene number in compact form (PRE 430 · S2).
  • Bypass state (green dot / 50% opacity + grayscale) is clear; document it. Selected block (amber ring) vs adjusted block (fill height) also reads well — keep.

7 · Accessibility Findings

  • Semantics & keyboard (blocking): every control is a div. Add role="button" tabIndex="0" + Enter/Space handling to all clickables (one shared helper), role="tablist/tab" on pane tabs and segmented controls, role="menu/menuitem" on context/ribbon menus, aria-pressed on toggles, aria-label on every icon-only button.
  • Focus: define one focus-visible ring (2-px accent, 2-px offset) and apply globally. Currently zero focus styles exist.
  • Escape & focus trap: global Escape closes (in order): context menu → ribbon dropdown → flyout/drawer/library → exits Edit Layout (with the same Done semantics). Trap focus inside flyouts/library while open; return focus to the trigger on close.
  • Drag-only features need parity: the right-click menu already exists — extend it on widgets with “Move to → Top left / Top right / Rail / Bottom / Floating / panel…” and “Size → Default / Compact / Mini”, and on panes with “Dock to → …”. This gives keyboard/switch users the full customization feature set without new UI concepts.
  • Sliders/knobs: grid swipe-adjust and editor knobs need role="slider" + arrow-key increments (±1, ⇧±10) once focusable, with aria-valuenow/text.
  • Reduced motion: wrap lzSlideR/lzRise/lzPulse/toast animation in @media (prefers-reduced-motion: no-preference); the pulse on live drop zones gets a static high-contrast border fallback.
  • Contrast: per §3.2 — one token change fixes ~40 call sites.

8 · Component-Level Recommendations

AxisWidget
Single chip/segment spec (§3.3); 44-px hit areas; hold-to-repeat on steppers (tempo, PRE arrows); promote .axtip to all icon-only widgets; add estW entries for every new widget kind (make the fallback log a dev warning — silent 120 px guesses break auto-fit).
Pane header
Consolidate the up-to-5 mini buttons (axis, tabs, collapse, dock-collapse, close) into: collapse + close visible, the rest in a kebab menu that merges with the existing right-click menu. Rename becomes a menu item with inline edit (drop window.prompt). Keep drag-to-dock on the whole header.
Widget Library
Good structure (Saved / Hidden / Placed / Hidden nav). Add: search field once list > 12; drag straight from Library onto the layout (tap-to-add currently dumps everything to top-right); group items by category (Status, Transport, FC, Params). Bottom sheet on tablet (M6).
Context menu
Extend with Move-to and Size submenus (a11y parity, §7); add to single widgets too (currently only groups/panels have one); keyboard navigable; clamp vertically as well as horizontally (currently only X is clamped — long menus can overflow the bottom edge).
Grid panel
Pane-relative sizing (R2); keep the 12×4 device matrix always; “Auto” must guarantee all rows visible at any pane size (scale down to map density before ever scrolling); empty-cell click opens Add-block palette (W7); document swipe/double-tap gestures in a first-run coach mark.
Block Editor
“Remove” (destructive, red) sits in the same row as Mute/Engaged with no confirmation — require confirm or provide undo toast; channel A/B/C/D selector is display-only in the shared editor element (hardcoded active A) — wire it; knobs need keyboard + double-click-to-type-value.
History (panel + drawer)
Two visual treatments of the same data — unify row spec (dot / label / time / restore affordance) and make rows interactive (W5). Panel rows currently have no hover state at all; drawer rows have hover but no action.
Toast
Queue (≤3), optional action slot, semantic tint (success teal / warning amber / destructive red edge), role="status" live region. Bottom-center is fine; keep clear of the FAB (currently both sit bottom-left/center and can overlap the Customize pill at narrow widths).

10 · Do-Not-Remove Constraint Check

Every recommendation is additive or reorganizational. Verified feature-by-feature:
  • All 14 dockable panels, tabs, splits, collapse (pane + dock + strips), cell resize, main-axis flip — unchanged; pane-header kebab relocates axis/tab buttons, it does not remove them.
  • Widget zones, grouping, density cycle, floating widgets, pinned params, edge-drop panel creation, Library save/load, layout presets, profiles — unchanged; auto-fit only adds shrink behavior.
  • Preset flyout, quick recall, scenes, view switch, FC bus controls, swipe-adjust, double-tap reset — kept; guards (dirty dialog, touch long-press arming) add a step only when data would otherwise be lost.
  • Legacy editorMode drawer/right/left/floating placements — kept as user-facing options; unification (R6) is internal: they become dock.place aliases, not removed capabilities.
  • One watch-item: making “Sets” open the Preset Browser replaces a no-op, so nothing is lost; the flyout remains on the preset widget.

11 · Final Implementation Brief

Order of work — each step independently shippable:
  1. Geometry correctness (P0): zone-aware widget auto-fit for tl/tc/tr/bottom/gridbar; grid pane sizes from its own pane rect; tablet region defaults as % of frame; fix detached nav-hide badges; anchor ribbon dropdowns to their triggers.
  2. Navigation truth (P0): Sets → Preset Browser dock set; Ctrl/Scn/Live/Setup → stub panels in main with honest empty states; second press toggles closed (FC pattern).
  3. State model (P1): dirty flag + Save widget states + recall guard dialog + ⌘S; connection status widget with 4 states; toast queue with action slot.
  4. Input layer (P1): shared focusable-button helper (role/tabindex/Enter/Space/focus ring), global Escape stack, ⌘Z/⇧⌘Z, ⌘K palette, context-menu Move-to/Size submenus, 44-px hit-area expansion, touch long-press arming for swipe-adjust.
  5. Internal unification (P1): gridDock/editorMode collapse into dock.place aliases; single segmented-control, chip, and tooltip components; History row spec + restore.
  6. Token pass (P2): type scale (kill <10 px), textfaint/textmuted contrast fix, 4-px spacing grid, radius trio, named accent tints, prefers-reduced-motion.
Rules during implementation: everything new is a Widget, Panel, or Parameter (CLAUDE.md); default panels keep their fixed internal layouts; panes stay absolutely positioned with hard clamps and no CSS transitions on pane geometry; nothing overlaps except tooltips/menus/modals; docking reflows neighbours, never paints over them; layouts persist per profile; empty zones hide outside Edit mode; no feature becomes harder to reach than it is today; match the existing palette — no new colors beyond named tints of the current ones.
